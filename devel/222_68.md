# [222_68] 聚光灯教程基础设施

## 如何测试

1. xmake config --tutorial=true
2. 清理缓存
3. xmake b stem ; xmake r stem
4. 查看首次启动的引导教程视觉效果

## 如何修改步骤

首次启动教程的步骤由外部配置文件定义：

+ `TeXmacs/plugins/tutorial/data/first-launch-tutorial.json`

该配置文件是一个 JSON 对象，描述一个教程 flow，当前支持的字段如下：

+ `flow-id`：教程流程 ID
+ `version`：教程版本号
+ `steps`：步骤数组
+ `id`：步骤唯一标识
+ `title`：步骤标题
+ `top-text`：正文上方文字，可为空
+ `media-path`：正文中间图片或 GIF 路径，可为空
+ `bottom-text`：正文下方文字，可为空
+ `on-enter`：进入该步骤时执行一次的 Scheme 命令，可为空
+ `target-id`：步骤要高亮的目标区域，可为空；为空时仅显示说明卡片，不绘制高亮区域
+ `placement`：说明卡片相对高亮区域的位置，可选值为 `auto / top / bottom / left / right`
+ `bubble-size`：说明卡片尺寸，可选值为 `small / medium / large`，未设置时默认为 `medium`
+ `offset-x`：说明卡片最终位置的水平偏移量，单位像素；右正左负
+ `offset-y`：说明卡片最终位置的垂直偏移量，单位像素；下正上负
+ `highlight-padding`：高亮区域向外扩展的边距
+ `skip-if-missing`：目标区域缺失时是否允许跳过该步骤

常见修改方式如下：

+ 调整步骤顺序：直接修改 `steps` 数组中的对象顺序
+ 修改步骤文案：修改 `title`、`top-text`、`bottom-text`
+ 进入时执行动作：修改 `on-enter`
+ 修改高亮目标：修改 `target-id`
+ 修改卡片位置：修改 `placement`
+ 修改卡片尺寸：修改 `bubble-size`
+ 微调卡片位置：修改 `offset-x`、`offset-y`
+ 修改高亮范围：修改 `highlight-padding`
+ 控制目标缺失行为：修改 `skip-if-missing`

如果 `target-id` 对应的是现有 Qt 控件，则需要确保该控件有稳定的 `objectName`，并在 `FirstLaunchTutorialController::buildRegistry()` 中注册。
如果目标不是单个稳定控件，而是一块逻辑区域，则应在 `buildRegistry()` 中新增 `registerRectProvider()`，先定义该区域的矩形，再在配置文件中引用对应的 `target-id`。

当前首次启动教程已使用的逻辑区域包括：

+ `mainWindowSafeArea`
+ `toolbarArea`
+ `editorArea`
+ `assistantEntry`

如果不通过配置文件修改，也可以在 `src/Plugins/Qt/qt_tutorial.cpp` 中修改 `FirstLaunchTutorialController::buildFallbackFlow()`；这是外部配置缺失或解析失败时的代码内回退配置。

## 2026/04/14 首次启动教程配置化增强

### What

当前首次启动教程在原有聚光灯基础设施上，新增了几项与配置文件直接相关的能力：

+ 当 `media-path` 为空时，不再渲染灰底空白图片区块
+ 当 `target-id` 为空时，可以只显示说明卡片，不绘制任何高亮区域
+ 步骤支持配置卡片尺寸 `bubble-size`
+ 步骤支持配置卡片偏移量 `offset-x` / `offset-y`
+ 首次启动教程第三步、第四步接入了 GIF 演示图
+ 第二步卡片位置从 `left` 调整为 `bottom`，避免默认布局下更容易贴边

### Why

前一版基础设施已经具备“高亮某块区域并跟随显示说明卡片”的能力，但在首次启动教程的实际使用中暴露出几个明显问题：

+ 没有图片的步骤仍然保留了浅灰色图片区占位
+ 欢迎页实际上不需要高亮任何区域，却被迫绑定一个大范围目标
+ 某些步骤需要更大的卡片去承载 GIF，而有些步骤只需要较小卡片
+ 仅依赖 `placement` 的四向布局还不够，部分步骤需要做像素级微调

这些问题都不应该通过继续硬编码 UI 分支来解决，更合适的方式是把能力下沉到教程配置模型中。

### How

+ `TutorialStepConfig` 中新增 `bubbleSize`、`offsetX`、`offsetY` 字段，默认值分别为 `medium`、`0`、`0`
+ `TutorialConfigLoader::loadFlow()` 读取步骤时，额外解析 `bubble-size`、`offset-x`、`offset-y`
+ `bubble-size` 当前支持三档：`small / medium / large`
+ `TutorialBubble::setStep()` 根据 `bubble-size` 切换卡片固定宽度和媒体区域尺寸：
  `small` 对应 `300` 宽和 `240x144` 媒体区
  `medium` 对应 `360` 宽和 `300x180` 媒体区
  `large` 对应 `440` 宽和 `380x228` 媒体区
+ `TutorialBubble::setStep()` 在 `media-path` 为空时显式隐藏 `m_mediaLabel`，避免空白占位
+ `TutorialEngine::showStep()` 对 `target-id` 为空的步骤直接走“仅展示卡片、清空高亮”的分支
+ `TutorialOverlay::bubbleRectForPlacement()` 在最终气泡坐标上叠加 `offset-x` / `offset-y`
+ `TeXmacs/plugins/tutorial/data/first-launch-tutorial.json` 当前已使用这些能力：
  第一页 `target-id` 置空，用作无高亮欢迎页
  第二页 `placement` 调整为 `bottom`
  第三页和第四页的 `bubble-size` 调整为 `large`
  第三页和第四页接入 `TeXmacs/misc/images/tutorial/test.gif`

## 2026/04/14 首次启动教程当前配置

### What

当前默认首次启动教程一共四步：

+ 第一步：欢迎页；无高亮；`medium`
+ 第二步：登录账户；高亮 `assistantEntry`；`medium`
+ 第三步：魔法粘贴；高亮 `editorArea`；`large`
+ 第四步：OCR 文字识别；高亮 `editorArea`；`large`

### How

这些步骤定义在：

+ `TeXmacs/plugins/tutorial/data/first-launch-tutorial.json`

当前配置示例可直接参考该文件中的以下字段组合：

+ `placement + bubble-size + offset-x + offset-y`
+ `target-id=""` 的无高亮欢迎页
+ `media-path` 指向相对路径 GIF 的演示步骤

## 2026/04/03 聚光灯教程基础设施

### What

当前实现提供了一套通用的聚光灯教程基础设施，用于在任意 Qt 主窗口上运行“遮罩 + 局部高亮 + 跟随式说明卡片”的引导流程。

基础设施由以下部分组成：

+ `TutorialFlowConfig` / `TutorialStepConfig`：教程流程与步骤的数据模型
+ `TutorialTargetRegistry`：目标区域注册表，负责注册控件和逻辑矩形
+ `TutorialConfigLoader`：外部配置加载器，负责从 JSON 文件读取教程 flow
+ `TutorialOverlay`：覆盖主窗口的遮罩层，负责全局禁点与局部高亮
+ `TutorialBubble`：步骤说明卡片，提供“上一步 / 下一步 / 完成”交互
+ `TutorialEngine`：通用教程引擎，负责运行教程流程
+ `FirstLaunchTutorialController`：首次启动教程接入层，负责在首次安装启动时加载并启动教程

首次启动教程只是这套基础设施的一个默认调用方，不再承担基础设施本身的定义职责。

### Why

教程系统的职责应当是“提供教程能力”，而不是“写死某一套首次启动步骤”。

因此当前实现将以下内容彼此分离：

+ 运行教程的能力
+ 注册目标控件和逻辑区域的能力
+ 步骤配置数据
+ 首次启动时机判断与完成状态持久化

这种拆分的好处是：

+ 教程引擎可以复用到其他引导场景，而不局限于首次启动
+ 步骤内容不需要硬编码在引擎中
+ 目标区域由调用方按需注册，而不是由引擎反向猜测
+ 首次启动判定、版本判断、偏好写入由调用方控制，不污染基础设施边界

### How

当前实现采用以下结构：

+ `src/Plugins/Qt/qt_tutorial.hpp` 中定义 `TutorialPlacement`、`TutorialFinishReason`、`TutorialStepConfig`、`TutorialFlowConfig`，作为教程数据模型
+ `src/Plugins/Qt/qt_tutorial.hpp` 与 `src/Plugins/Qt/qt_tutorial.cpp` 中实现 `TutorialTargetRegistry`，对外暴露 `registerWidget()` 与 `registerRectProvider()` 能力
+ `src/Plugins/Qt/qt_tutorial.hpp` 与 `src/Plugins/Qt/qt_tutorial.cpp` 中实现 `TutorialConfigLoader`，用于从 Scheme 文件加载教程 flow
+ `src/Plugins/Qt/qt_tutorial.hpp` 与 `src/Plugins/Qt/qt_tutorial.cpp` 中实现 `TutorialOverlay` 与 `TutorialBubble`，分别负责遮罩绘制和步骤卡片交互
+ `src/Plugins/Qt/qt_tutorial.hpp` 与 `src/Plugins/Qt/qt_tutorial.cpp` 中实现 `TutorialEngine`，作为通用教程引擎，统一负责启动、停止、步骤切换、失败重试、缺失目标回退、窗口 resize/move 跟随，以及 `finished / stepChanged` 信号发出
+ `TutorialEngine` 启动时显式接收 `QMainWindow*`、`TutorialFlowConfig`、`TutorialTargetRegistry`，不再持有硬编码步骤、不再直接判断首次启动、不再直接读写用户偏好
+ `src/Plugins/Qt/qt_tutorial.hpp` 与 `src/Plugins/Qt/qt_tutorial.cpp` 中实现 `FirstLaunchTutorialController`，作为首次启动教程的默认接入层，负责首次安装判定、读取外部配置、构建注册表、监听引擎完成事件，并记录 completed/skipped 版本
+ `TeXmacs/plugins/tutorial/data/first-launch-tutorial.json` 作为首次启动教程的默认外部配置文件
+ `FirstLaunchTutorialController::buildFallbackFlow()` 提供代码内 fallback flow，保证配置文件缺失或解析失败时教程仍可运行
+ `QTMWindow.cpp` 的 `showEvent()` 中调用 `FirstLaunchTutorialController::instance()->maybeStartForMainWindow(this)`，将首次启动教程接入主窗口生命周期

## 2026/04/03 教程目标回退与导航行为

### What

当前基础设施对“目标控件启动早期尚不可见”这一情况提供了统一处理策略：

+ 目标解析失败时先短暂重试
+ 重试后仍失败时，按当前导航方向查找下一可用步骤
+ 若步骤显式配置为不跳过，则保留说明卡片并清空高亮区域

同时，首次启动教程中的工具栏和编辑区使用逻辑区域回退，避免依赖某个单一控件必须始终可见。

### Why

Qt 主窗口的部分区域在启动早期可能尚未创建完毕或尚未变为可见状态。
如果教程系统只依赖单个控件，步骤切换时就容易出现卡住、跨步或回退方向错乱。

因此教程基础设施需要同时具备：

+ 目标解析重试能力
+ 方向感知的步骤回退能力
+ 逻辑区域回退能力

### How

+ `TutorialEngine::showStep()` 内置重试逻辑，并在重试耗尽后按当前导航方向决定是否跳到下一个可用步骤
+ `TutorialEngine::showNextAvailableStep()` 统一处理向前和向后导航
+ `FirstLaunchTutorialController::buildRegistry()` 中为 `toolbarArea`、`editorArea`、`assistantEntry` 提供逻辑区域 provider
+ `TutorialOverlay::paintEvent()` 使用较细描边与较轻发光效果，降低高亮区域对正文内容的压迫感
